## Overview

This file is the master work-breakdown for building the generator across many short-lived Claude Code sessions. The full implementation is too large for one session (context overflow risk), so it is split into **25 discrete steps**. Each step becomes its own `specs/step-N-<slug>.md` (written *before* that step's session starts), enters Claude Code in plan mode for alignment, and is then implemented in a single fresh session with a clean context window.

**Per-session flow:**
1. Read the relevant spec + the reference files it names (NOT the whole repo).
2. Enter plan mode, confirm plan with user.
3. Exit plan mode, implement, verify exit criteria.
4. Commit, update any ref file if a spec conflict was discovered at the source (per PRD §10 priority), and leave a short handoff note for the next session.

Each step targets **S scope (1 session)** unless explicitly marked M. Anything that looked L has already been split in this file.

## Seed Data Authoring Convention

Tier 0 seed modules (`seed_data/*.py` — Steps 6, 7, 8) are **handwritten Python dict literals** produced by Claude Code in-session. They are not extracted from a data file and not generated by a script.

**Reading discipline per seed session:**

| Read (required) | Purpose |
|-----------------|---------|
| `references/07_mvp-schema-reference.md` | DDL slice for the in-scope tables — column names, types, nullability, composite PKs |
| `references/02_data-mapping-reference.md` Step 3 | Constrained code values and literal-match rows (e.g. `FROZEN`/`Frozen Status`/`Frozen`, `Rate Feature`, `Original Loan Term`, the 4 ORG name types, the 4 AGREEMENT_FEATURE_ROLE_TYPE roles) |
| `references/05_architect-qa.md` | Only the Q's that touch the current domain |

**Do not read:**
- `references/01_schema-reference.md` — supplementary; open only if `07` is ambiguous for a specific column
- `references/06_supporting-enrichments.md` — distributions; used by Step 4 UniverseBuilder, irrelevant for seed tables
- `resources/CIF_FSDM_Mapping_MASTER.xlsx` and `resources/iDM_MDM_tables_DDLs.xlsx` — already distilled into the references above

**Escalation.** If a constrained code appears in `02` without a complete enumeration, or `07` has a column whose legal values are unclear, flag it via Handoff Protocol §2 (update the upstream reference or leave a `⚠️ Conflict` block in the spec) — do not silently improvise. The Excels are the ground truth of last resort and should be consulted by the architect/user, not silently re-read per session.

**Output contract.** Each seed module exposes `get_<domain>_tables() -> Dict[str, pd.DataFrame]`. Step 8 wires them into `ctx.tables` via `Tier0Lookups.generate(ctx)` with DI column stamping applied uniformly.

### Step 1: Project Scaffolding & Config
- **Spec:** `specs/step-1-scaffolding.md`
- **Depends on:** none
- **Produces:**
  - Directory tree per `mvp-tool-design.md` §3 (`config/`, `seed_data/`, `registry/`, `generators/`, `utils/`, `output/`)

- **Reads from:** `PRD.md` §4, §7; `mvp-tool-design.md` §1, §3
- **Exit criteria:**
  - `python -c "import config.settings, config.code_values, config.distributions"` succeeds
  - Folder tree matches design doc exactly
  - All constants from `mvp-tool-design.md` §1 and §7 are represented with correct values

### Step 5: Output Writer
- **Spec:** `specs/step-5-writer.md`
- **Depends on:** Step 2
- **Produces:**
  - `output/writer.py` — `Writer.write_all(tables)` walking `ctx.tables`, writing one CSV per `'Schema.TABLE'` into the correct subdir

- **Exit criteria:**

  - `PARTY_INTERRACTION_EVENT.csv` filename preserves the typo when that table is written
  - `GEOSPATIAL` is explicitly skipped (no file written)
- **Scope:** S

## Handoff Protocol

At the end of every session:

1. **Commit the work.** One commit per step (e.g. `feat(step-6): tier 0a seed data — agreement/financial/feature`). Use a HEREDOC commit message that summarizes the step's deliverables in one line. Do NOT force-push or amend prior commits.

2. **Record spec conflicts, if any.** If the step uncovered a contradiction between the spec and a priority-1 reference (`02_data-mapping-reference.md` or the master Excel), *do not silently work around it.* Update the upstream reference if the architect has been consulted, or add a `⚠️ Conflict` block to the spec with a pointer to where the decision needs to be made. Follow PRD §10 priority.

3. **Leave a handoff note.** At the bottom of the step's spec, append a `## Session Notes` section with:
   - What was actually built vs. what the spec said
   - Any deferred items (with a TODO or follow-up issue)
   - One-line guidance for the next session (e.g., "Step 7 can start now — Tier 0a seed loader is stable")

4. **Do not update other steps' specs retroactively** unless a discovered fact changes their exit criteria materially. In that case, append a dated note under the affected step's criteria — don't rewrite history.

5. **Run the partial smoke test where possible.** From Step 8 onward, `python main.py` should execute the tiers built so far (tiers not yet implemented stay as no-ops in the orchestrator). The validator stays disabled until Step 24; until then, rely on per-tier exit criteria only.

6. **Context hygiene for the next session.** The next session should start fresh and only read: (a) its own spec, (b) the specific reference files the spec names, and (c) a glance at the immediately prior step's output code — never re-read the full reference set. `MEMORY.md` already points at project-level decisions; trust it.
